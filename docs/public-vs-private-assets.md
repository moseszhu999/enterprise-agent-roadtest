# Public vs Private Asset Strategy

This repository is public by design, but the public layer and the private asset layer must be kept separate.

The public repository is not the moat.

The public repository is a signal that the category is real and inspectable.

## Public layer

The public layer should include:

- category positioning,
- synthetic roadtest samples,
- basic scoring rubrics,
- simple failure taxonomies,
- privacy notes,
- example Agent outputs,
- lightweight evaluation scripts,
- non-confidential reports,
- public articles and discussion drafts.

The purpose of the public layer is to show:

1. the problem is concrete,
2. the approach is practical,
3. the work is not only a slogan,
4. the roadtest format can be inspected,
5. the project is open to critique.

## Private asset layer

The private layer should include:

- richer synthetic task libraries,
- industry-specific roadtests,
- real Agent outputs from actual evaluations,
- human evaluator notes,
- calibration records,
- detailed failure labels,
- customer-ready report templates,
- partner feedback,
- pricing and packaging notes,
- outreach lists,
- customer-specific scenarios,
- any customer-provided or partner-provided materials.

These should not be placed in the public repository.

## What should never be public

Do not publish:

- customer business data,
- real procurement records,
- real supplier names,
- ERP exports,
- contracts,
- private Agent outputs,
- customer PoC reports,
- private partner feedback,
- commercial scoring logic that is not intended to be copied,
- sales pipeline information.

## Why this separation matters

The public layer creates trust and category visibility.

The private layer builds accumulated advantage.

A single public test can be copied.

A continuously maintained library of roadtests, Agent outputs, failure labels, calibration notes, and customer reports is harder to copy.

## Strategic position

This project should not pretend that the first roadtest is technically defensible by itself.

The early advantage is:

- naming the category,
- showing a working sample,
- building a repeatable evaluation language,
- attracting high-quality critique,
- accumulating real failure evidence over time.

## Rule of thumb

Publish enough to make the idea credible.

Keep enough private to make the work compound.

## Public repository role

The public repository should function as:

- a showroom,
- a research notebook,
- a category signal,
- a proof-of-work artifact,
- a safe place for synthetic examples.

It should not function as the full commercial product.

## Private asset role

The private layer should become:

- the roadtest library,
- the failure evidence library,
- the calibration history,
- the customer report factory,
- the basis for future services or software.
